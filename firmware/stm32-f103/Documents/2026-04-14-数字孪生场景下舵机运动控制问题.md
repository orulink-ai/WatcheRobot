## 2. 观察到的现象

### 2.1 定位不准

- 舵机有时没有到达指定位置
- 实际位置与目标位置存在偏差
- 某些关键帧似乎被"跳过"了

### 2.2 运行卡顿

- 运动过程中一卡一卡的
- 不像 Blender 中那样流畅
- 运动节奏不连续

### 2.3 抖动

- 特定情况下出现明显抖动
- 舵机发出异常声响
- 机械结构不稳定

---

## 3. 根本原因分析

### 3.1 Blender 是"数学空间"，舵机是"物理空间"

| 维度 | Blender 动画 | 物理舵机 |
|------|--------------|---------|
| 位置变化 | 瞬时突变 | 渐变响应 |
| 时间模型 | 理想时间线 | 实际有延迟 |
| 扭矩假设 | 无限扭矩 | 扭矩有限 |
| 惯性考虑 | 无惯性 | 机械惯性 |

### 3.2 问题 1：定位不准

**原因：舵机没收到或没执行完指令，就收到新指令**

```
时序问题（假设串口通信）：

Blender:    发 [0°]  ─── 发 [90°]  ─── 发 [45°]
            0ms       100ms        200ms

舵机接收:   收 [0°]  ─── 收 [90°]  ─── 收 [45°]
            5ms       105ms        205ms

实际执行:   收到 90° 时正在走，但新指令 45° 又来了
           舵机：就地停下，转去 90°...

结果: 根本没到达过 90°，直接去了 45°
```

### 3.3 问题 2：运行卡顿

**原因：目标值突变，但舵机无法瞬时响应**

```
错误方式：直接设置目标位置
Servo_SetAngle(hServo, targetAngle);  // 舵机以最大速度冲过去

当关键帧变化太快时：
目标: 0° → 90° → 45° → 135° → 60° → ...
      ↑ 每个目标都在舵机还没到达时就换下一个了
      → 舵机一直处于"还没到又想走"的状态
      → 卡顿！
```

### 3.4 问题 3：抖动

**原因：舵机在两个位置之间反复横跳（PID 振荡）**

```
目标值快速变化时，舵机内部 PID 控制器的响应：

实际位置:  ───► 80° ──────────► 90°
            ↗  收到 85°   ↘ 收到 95°  ↗ 收到 88°
            ↓  转头      ↓  转头    ↓  转头
实际轨迹:  60°↗ 70° ↘ 75° ↗ 82° ↘ 78° ↗ 85°...
           ─────────────────────────────────────
                    抖动！震荡！失控！
```

### 3.5 核心问题：缺少"速度规划"

**Blender 关键帧** → **瞬时位置指令** → **舵机无法执行**

```
对比：

无速度规划（直接给目标）：
        ▲
90°    │    ────
        │  ↗
0° ────│─
        └──────────────► 时间
          卡顿！

有速度规划（梯形曲线）：
        ▲
90°    │      ╱╲
        │    ╱    ╲
        │  ╱        ╲
0° ────│╱
        └──────────────► 时间
          平滑！

有速度规划（S曲线）：
        ▲
90°    │     ╭──╮
        │   ╭╯    ╰╮
        │  ╭╯        ╰╮
0° ────│╯
        └──────────────► 时间
          更平滑！
```

## 4. 解决方案

### 4.1 方案一：STM32 端实现速度规划（推荐）

**在舵机驱动层实现梯形或 S 曲线速度规划**

```c
typedef struct {
    uint16_t currentPulse;       // 当前位置 (μs)
    uint16_t targetPulse;        // 目标位置 (μs)
    uint16_t speed;              // 速度 (μs/ms 或 °/s)
    uint8_t  state;              // 状态: IDLE/MOVING/ARRIVED
} Servo_MotionPlanTypeDef;

// 梯形速度规划
void Servo_MoveToAngle(Servo_HandleTypeDef hServo,
                        uint16_t targetAngle,
                        uint16_t speedDegPerSec)
{
    // 1. 计算当前位置到目标位置的距离
    // 2. 根据梯形/S曲线规划速度曲线
    // 3. 在 Servo_Update() 中逐步执行
}
```

**优点**：
- 舵机自己控制速度，物理行为平滑
- 不依赖上位机通信时序
- 可复用

**缺点**：
- 需要修改舵机驱动层
- 需要调试速度规划参数

### 4.2 方案二：关键帧插值

**在 PC 端（Blender 侧）对关键帧做时间插值**

```
原始关键帧（太快）：
Blender:  0° @ 0ms ──────► 90° @ 500ms
             ↓
插值后（平滑）：
        0°→18°→36°→54°→72°→90° (每50ms一个点)
```

**优点**：
- 不修改 STM32 代码
- 可以实现复杂的速度曲线

**缺点**：
- PC 端需要额外处理
- 插值算法需要匹配舵机特性

### 4.3 方案三：关键帧队列缓冲

**在舵机端缓冲关键帧，先进先出执行**

```c
#define KEYFRAME_QUEUE_SIZE 16

typedef struct {
    uint16_t targetAngles[KEYFRAME_QUEUE_SIZE];  // 缓冲队列
    uint8_t  head;
    uint8_t  tail;
    uint8_t  count;
} KeyFrameQueue;

// 添加关键帧到队列
void Servo_PushKeyFrame(Servo_HandleTypeDef hServo, uint16_t angle);

// 在 Servo_Update() 中执行
void Servo_Update(void) {
    // 队列非空 且 当前运动完成 → 弹出下一个关键帧
}
```

**优点**：
- 可以吸收突发的大量关键帧
- 舵机按自己的节奏执行

**缺点**：
- 引入额外延迟
- 需要处理队列溢出

### 4.4 方案对比

| 方案 | 复杂度 | 延迟 | 流畅度 | 推荐场景 |
|------|--------|------|--------|---------|
| STM32 速度规划 | 中 | 低 | 高 | **首选**，通用场景 |
| 关键帧插值 | 低 | 低 | 取决于插值算法 | 上位机计算资源充足时 |
| 关键帧队列 | 低 | 中 | 中 | 突发大量关键帧时 |

## 5. 实施建议

### 5.1 优先级

1. **立即可行**：方案三（关键帧队列）— 最简单
2. **长期方案**：方案一（速度规划）— 最完善

### 5.2 调试建议

1. 先用示波器或逻辑分析仪观察 PWM 波形
2. 检查关键帧发送频率 vs 舵机响应时间
3. 测量关键帧之间的实际时间间隔

### 5.3 参数估算

假设：
- 舵机转速：约 200°/s（全速）
- 关键帧频率：50ms 一个
- 关键帧角度差：平均 30°

计算：
- 舵机全速运行 30° 需要：30° / 200°/s = 150ms
- 关键帧间隔：50ms
- **结论**：每个关键帧舵机只能完成 1/3 的路程
