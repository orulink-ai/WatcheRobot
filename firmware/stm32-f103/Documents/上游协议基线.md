> 公开阅读提示：本文是 STM32F103 后续开发参考，用于说明历史 baseline 与引用边界，不是当前公开协议真源。公开协议总览见 [通信协议总览](../../COMMUNICATION_PROTOCOLS_zh.md)。

## 1. 固定 baseline

当前 `watcherobot_stm32` 仓库后续统一引用：

```text
Protocol baseline: orulink-ai/WatcheRobot@proto/v2.0.0-stm32-align-20260416
```

解析到的 upstream commit 固定为：

```text
cedfaee49169d61450dcb8d6576a96bf6c012996
```

除非用户明确更新 baseline，否则本仓内所有开发文档、分支规划、PR 描述、联调记录都默认引用这条 baseline。

## 3. 本仓引用规则

本仓的规则固定如下：

- 不再维护第二份独立协议字段表
- 不在本仓重新定义消息号、flags、`ACK/NACK/FAULT` 语义
- 不在本仓复制完整 payload 结构作为“本地真源”
- 本仓只记录：
  - 正在引用哪个 upstream baseline
  - 当前本地计划实现到哪一个阶段
  - 当前本地与 baseline 的差距是什么

如果需要引用协议关键点，应采用“引用 + 摘要”的方式，而不是在本仓重新写一份完整协议表。

## 4. 当前本地实现范围判断

按当前仓库已提交状态判断：

- 本地仓仍处于 pre-v2 demo 阶段
- 已有 `STM32F103C8Tx`、`72MHz`、`USART2@921600` 的基础工程条件
- 已形成 `USART2 RX DMA + IDLE` bring-up 链路
- 尚未形成 `COBS + CRC16 + 0x00 delimiter` 接收路径
- 尚未形成二进制 frame 编解码
- 尚未实现 `HELLO_REQ -> ACK -> HELLO_RSP`
- 尚未实现 `ACK / NACK / FAULT / MOTION_DONE / LED_DONE / SENSOR` 协议闭环

因此，本地下一阶段应从协议核心开始，而不是提前切到 motion、led、sensor 功能扩展。

## 5. 当前本地计划实现范围

当前本地开发计划的阶段顺序固定为：

1. P0 文档对齐
2. P1 工程与 UART2 bring-up
3. P2 协议核心
4. P3 motion
5. P4 led
6. P5 sensors
7. P6 recovery
8. P7 bench / HIL / compatibility 收口

当前仓库优先推进的最近范围是：

- P1 工程与 `USART2 RX DMA + IDLE` 收口
- P2 `HELLO_REQ -> ACK -> HELLO_RSP` 最小闭环
