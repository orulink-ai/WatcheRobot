# 分支规划

## 1. 主线规则

- `dev` 是唯一集成主线
- 所有后续功能分支都从最新 `dev` 切出
- 所有后续功能分支都回合到 `dev`
- 不在本仓维护多条长期并行的集成主线

## 2. 分支命名规则

- 文档整理：`docs/...`
- 功能实现：`feat/...`
- 如需修复单点缺陷：`fix/...`

当前固定分支命名如下：

- `docs/stm32-v2-baseline-20260416`
- `feat/stm32-uart2-bringup`
- `feat/stm32-protocol-rx`
- `feat/stm32-motion`
- `feat/stm32-led`
- `feat/stm32-sensors`
- `feat/stm32-recovery-bench`

## 3. 分支依赖顺序

后续依赖顺序固定为：

```text
dev
  -> docs/stm32-v2-baseline-20260416
  -> feat/stm32-uart2-bringup
  -> feat/stm32-protocol-rx
  -> feat/stm32-motion
  -> feat/stm32-led
  -> feat/stm32-sensors
  -> feat/stm32-recovery-bench
```

说明：

- `feat/stm32-recovery-bench` 放在 `motion / led / sensors` 之后
- 不提前并行推进恢复链路与联调收口

## 4. 各分支目标与完成标准

### `docs/stm32-v2-baseline-20260416`

- 目标
  - 建立 `Documents/` 主入口
  - 固定 upstream baseline
  - 固定本仓分支规划与后续任务拆解
  - 把 README 从旧 demo 入口改成 v2 文档入口
- 进入条件
  - `dev` 为最新可用文档状态
- 退出条件
  - README、baseline、branch plan、development tasks 口径统一
  - 旧文档都有“历史/现状说明”横幅
- 合并目标
  - `dev`

### `feat/stm32-uart2-bringup`

- 目标
  - 明确 CubeMX/工程入口口径
  - 明确 `USART2 RX DMA + IDLE` 的目标配置
  - 明确保留 `USART1` 调试日志
  - 明确 VS Code 的 `build / flash / debug` 任务需求
- 进入条件
  - baseline 与文档入口已固定
- 退出条件
  - bring-up 所需配置、任务入口和完成标准都已说清楚
  - 注意：退出条件是文档和实现范围清晰，不代表代码已完成
- 合并目标
  - `dev`

### `feat/stm32-protocol-rx`

- 目标
  - 落定协议骨架范围
  - 明确环形缓冲、`0x00` 分帧、COBS、CRC16、frame parser/builder
  - 明确 `ACK/NACK/FAULT`
  - 明确 `HELLO_REQ -> ACK -> HELLO_RSP` 最小闭环
- 进入条件
  - bring-up 阶段的 UART2 接收链路目标已定
- 退出条件
  - 协议骨架范围和测试口径无歧义
  - 文档中明确禁止按旧口径理解成“直接请求响应”
- 合并目标
  - `dev`

### `feat/stm32-motion`

- 目标
  - `SERVO_MOVE`
  - `SERVO_STOP`
  - `MOTION_DONE`
  - motion fault 路线
- 进入条件
  - `HELLO / ACK / NACK / FAULT` 最小闭环已定义清楚
- 退出条件
  - 动作命令受理、执行完成与故障语义清晰
  - 不再把“上电自动往复”当作主线行为
- 合并目标
  - `dev`

### `feat/stm32-led`

- 目标
  - `LED_SET_STATIC`
  - `LED_SET_EFFECT`
  - `LED_OFF`
  - `LED_DONE`
- 进入条件
  - motion 路线和控制命令 ACK/DONE 语义已固定
- 退出条件
  - LED 命令、完成回执与 fault 语义清晰
- 合并目标
  - `dev`

### `feat/stm32-sensors`

- 目标
  - `TOUCH_EVENT`
  - `IMU_STATE`
  - `MAG_STATE`
  - `MAG_EVENT`
  - `SENSOR_HEALTH`
- 进入条件
  - control 类消息的协议处理路径已明确
- 退出条件
  - 状态流采用 `latest-state-wins`
  - 控制事件优先于状态遥测
- 合并目标
  - `dev`

### `feat/stm32-recovery-bench`

- 目标
  - `SNAPSHOT_REQ / SNAPSHOT_RSP`
  - 恢复路径与 safe-default baseline
  - bench / HIL / compatibility 收口
- 进入条件
  - motion / led / sensors 路线已形成完整文档化范围
- 退出条件
  - 恢复路径、首轮 bench 顺序、联调记录模板和回填位置清晰
- 合并目标
  - `dev`

## 5. PR 固定要求

每个 PR 描述都必须附带：

```text
Protocol baseline: Ro-In-AI/WatcheRobot-Firmware@proto/v2.0.0-stm32-align-20260416
```

建议同时附带：

- 本分支目标
- 当前完成范围
- 未覆盖范围
- 与上游 baseline 的对应阶段

## 6. 不允许的文档漂移

后续分支文档中不应出现以下情况：

- 在本仓重新定义完整协议字段表
- 在本仓复制一份“本地协议真源”
- 把 `USART1 CLI + 自动往复舵机` 继续描述成 v2 主线行为
- 把 `HELLO_REQ` 误写成“无需 ACK、直接等待 HELLO_RSP”
