# 开发任务拆解

> 公开阅读提示：本文保留为 STM32F103 阶段开发参考，不是当前公开协议真源或 Release 承诺。公开协议总览见 [通信协议总览](../../COMMUNICATION_PROTOCOLS_zh.md)。

## 1. 总体说明

本文记录 `watcherobot_stm32` 的完整 v2 路线，统一对应：

```text
Protocol baseline: orulink-ai/WatcheRobot@proto/v2.0.0-stm32-align-20260416
```

本文只做任务拆解和阶段口径，不复制完整协议字段表。所有消息号、flags、payload 与 `ACK/NACK/FAULT` 语义统一以上游真源为准。

## 2. 阶段顺序

```text
P0 文档对齐
P1 工程与串口 bring-up
P2 协议核心
P3 Motion
P4 LED
P5 Sensors
P6 Recovery
P7 Bench / HIL / Compatibility
```

分支顺序必须与上述阶段顺序一致，不提前并行切入恢复与 bench 收口。

## 4. 当前优先级

当前仓库最近优先级固定为：

1. P0 文档对齐
2. P1 工程与 `USART2 RX DMA + IDLE`
3. P2 `HELLO_REQ -> ACK -> HELLO_RSP` 最小闭环

在上述范围跑通前，不应把实现重点切到：

- motion 业务扩展
- LED 业务扩展
- sensor 全量上报
- 恢复与 bench 收口

## 5. 首轮 bench 固定检查顺序

后续 bench 验证建议固定按以下顺序进行：

1. 链路启动
   - ESP32 发出 `HELLO_REQ`
   - STM32 返回 `ACK`
   - STM32 返回 `HELLO_RSP`
2. 动作链路
   - `SERVO_MOVE`
   - `ACK`
   - `MOTION_DONE`
3. 灯效链路
   - `LED_SET_STATIC` 或 `LED_SET_EFFECT`
   - `ACK`
   - `LED_DONE`
4. 传感器链路
   - `TOUCH_EVENT`
   - `IMU_STATE`
   - `MAG_STATE`
5. 恢复链路
   - 断连或复位
   - 恢复
   - 再次验证最小闭环
